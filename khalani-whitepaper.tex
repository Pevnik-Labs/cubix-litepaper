\section{A deeper dive into Axi}

Axi (pronounced ``Axee'') is the smart contract language of Khalani. It is a purely functional programming language with a strong, static, linear type system and support for formal reasoning about programs in classical higher-order logic. Among mainstream programming languages, it is most similar to Haskell (in spirit) and OCaml (syntactically). It was also heavily inspired by proof assistants based on Type Theory, like Coq, Lean and Agda.

In this appendix we will have a brief, not too technical look at the philosophy and language features of Axi, together with a few simple code examples.

\subsection{Philosophy}

Axi's architecture is unique in that the language is split into two strictly separate layers: the programming layer and the logic layer.

\textbf{The Programming Layer} of Axi is a purely functional language based on System F with algebraic data types and a resource-aware type system. It aims to make programming pleasant and comprehending smart contract code easy, while providing essential guarantees to all on-chain code:

\begin{itemize}
  \item \textbf{Type safety}. Well-typed programs can't go wrong, i.e. they can't crash (unless they run out of gas), can't perform nonsensical operations like adding a number and a string, and can't result in any other runtime type error.
  \item \textbf{Memory safety}. Well-typed programs can't result in memory errors, like writing to a random memory location, nor can they have memory leaks.
  \item \textbf{Resource safety}. Well-typed programs can't access resources they don't have permission to, can't delete tokens that are not supposed to be deletable and can't create money out of thin air.
\end{itemize}

\textbf{The Logic Layer} of Axi is a language for reasoning about Axi programs in classical higher-order logic. It allows expressing arbitrarily complex mathematical properties, thus giving programmers the ability to precisely specify the business logic and behaviour of their smart contracts. These specifications can then be either tested with random inputs before deployment, checked for concrete inputs at runtime, or proved for all possible inputs using logic. The last of these, proving, provides guarantees much more powerful than those of the type system alone, while being as credible and trustworthy as the type system. The use of the logic layer is entirely optional, i.e. programmers are never forced to engage with any part of it, although competition will encourage programmers to do so in the hopes of outcompeting other programmers' smart contracts by providing stronger guarantees.

\textbf{Strict separation}. Logic can reason about programs, but programs cannot manipulate logical entities, like theorems or proofs, at runtime. This one-way interaction keeps the language relatively simple, while still making it possible to express almost all functional programming idioms and all possible mathematical properties and their proofs.

\textbf{Guarantees all the way down.} We not only want Axi to provide strong guarantees about its programs, but we also want to provide strong guarantees about Axi itself. We plan to use the Coq proof assistant to formally verify significant portions of Axi's metatheory, including the aforementioned properties of type and resource safety, as well as the consistency of Axi's logic. We are also considering formal verification of the reference implementation of Axi. The goal of these efforts is to ensure that the guarantees that Axi provides are trustworthy and will stay so forever, as long as the consensus protocol is followed.


\subsection{Language features}

In this section, we will take a closer look at Axi's language features.

\subsubsection{The Programming Layer}

Axi is, at its core, a purely functional programming language descending from the ML family. Thus, it supports all the usual features of such languages:

\begin{itemize}
  \item \textbf{Purity amd immutability}. In Axi, all values are pure and immutable -- they cannot be null, cannot throw exceptions and once created, they cannot be changed. This facilitates reasoning about and comprehending code, thus rendering Axi smart contracts transparent and auditable.
  \item \textbf{First-class, higher-order and anonymous functions}. Functions are first class citizens. They can be passed as arguments to other functions, returned from other functions as results and created on the fly. Axi encourages a programming style in which data is processed by pipelines built from basic building blocks like mapping, reducing, filtering and so on.
  \item \textbf{Algebraic data types and exhaustive pattern-matching}. Instead of clunky boolean flags and cascades of ifs, Axi makes it easy to model the problem domain with disjoint unions and define functions on them by pattern matching and explicitly handling all possible cases.
  \item \textbf{Strong static type system}. Axi's type system makes it easy to work with high-level abstractions, like merkle trees or state machines, instead of low-level primitives, like integers or byte arrays. It is lightweight, not placing too many demands on the programmer, but still catches the most egregious errors -- it can easily protect programmers from confusing with each other such objects as an integer, a string, an address, a transaction hash, a token amount and a cryptographic secret.
  \item \textbf{Polymorphism and interfaces}. Axi's type system features polymorphism (also known as ``generics'') as well as interfaces inspired by Haskell's type classes. These features make it easy to implement data structures and functions that work for all types that support a certain behaviour, for example a hash map that works for all types of ERC-20-compliant tokens bridged from an external blockchain.
\end{itemize}

However, these are only the standard features that plenty of functional programming languages have. The killer feature, as far as the programming layer is concerned, are linear types, which allow the type system to enforce correct resource usage discipline. This aspect of Axi's type system was inspired by linear logic and its cousin, Quantitative Type Theory, and is an improvement upon an analogous feature of the type system of Move, the programming language of the Sui and Aptos blockchains.

In an Axi program, every variable is assigned its \textit{quantity}, which can be either: zero, one, at most one, at least one, unrestricted. The variable's quantity determines how many times it can be used in the program, with too many or too few uses resulting in a type error. For example, the typical variable holding an NFT will have a quantity of one, forcing the program to do something with the NFT before it terminates: maybe send the NFT to the transaction's originator, maybe trade it for something else and handle that something else properly according to its quantity, or maybe send it to somebody as a gift. What the program cannot do in such a situation is forget the NFT and do nothing with it, effectively making it ``lost'' in the guts of the chain, nor copy the NFT and have two instead of just the original one.

To make programmers' life easier, every type is also assigned its ``default'' quantity. Every variable will be assigned this quantity unless the programmer decides otherwise. An important piece of terminology: a type whose default quantity is unrestricted is called a \textit{data type}, whereas a type whose default quantity is either one, at least one or at most one, is called a \textit{resource type}. There are no types whose default quantity is zero. Elements of data types are called data, whereas elements of resource types are called resources.

\subsubsection{The Logic Layer}

Axi's logic layer consists of typed classical higher-order logic with equality whose domains of discourse are the types from the programming layer. This means the logic consists of:

\begin{itemize}
  \item \textbf{Propositional connectives and constants}. Standard propositional connectives ($\Longrightarrow$, $\land$, $\lor$, $\Longleftrightarrow$, $\sim$) and constants ($\top$, $\bot$) can be used to build up complex propositions from simpler ones.
  \item \textbf{Quantification over terms and types}. Axi supports universal and existential quantification. We can quantify over terms of a given type ($\forall x : A, \dots$) as well as over types themselves ($\forall A : \texttt{Type}, \dots$), no matter whether we're dealing with data, resources or programs.
  \item \textbf{Higher-order quantification}. We can also quantify over logical entities like propositions ($\forall P : \texttt{Prop}, \dots$), predicates ($\forall P : A \rightarrow \texttt{Prop}, \dots$), relations ($\forall R : A \rightarrow A \rightarrow \texttt{Prop}, \dots$), logical connectives ($\forall C : \texttt{Prop} \rightarrow \texttt{Prop} \rightarrow \texttt{Prop}, \dots$) or even quantifiers themselves ($\forall Q : (A \rightarrow \texttt{Prop}) \rightarrow \texttt{Prop}, \dots$). This ability of logic to talk not only about the underlying programming language, but also about itself, gives it great expressive power and allows it to make use of all of mathematics when reasoning about the behaviour of programs.
  \item \textbf{Equality}. The built-in equality relation ($=$) makes it possible to precisely reason about programs' input-output behaviour.
  \item \textbf{Classical principles}. Unlike many other type-theoretical systems, we assume the Law of Excluded Middle as well as the Axiom of Choice, which makes the logic classical and thus intimately familiar to all mathematicians.
\end{itemize}

Proving is carried out in a proof language based on the Curry-Howard correspondence: proofs of implications and universal quantifiers are like functions, proofs of existential quantifiers and conjunctions are like pairs, proofs of disjunctions are like disjoint union constructors, etc. Negation and biconditional are handled as in intuitionistic logic - instead of being primitives, negation ($\sim P$) is a shorthand for an implication into falsehood ($P \implies \bot$), whereas biconditional ($P \iff Q$) is a shorthand for a conjunction of two implications ($(P \implies Q) \land (Q \implies P)$).

One novel feature that distinguishes our proof language is that the proofs can be presented in two styles: either in a \textit{proof term style}, similar to how proofs look in Agda and Idris (as well as Coq's and Lean's manually written proof terms), or in a \textit{tactic style}, similar to how proofs usually look in Coq or Lean. The difference between Axi and languages that also support both styles, like Coq and Lean, is that in these languages each style is its own separate proof language, whereas in Axi there is a single proof language which, thanks to a clever parser, can be formatted either like terms or like tactics. This feature will make it easier to accomodate computer scientists and mathematicians familiar with either of the aforementioned languages.

Equality is handled as in Martin-Löf Type Theory - proved by reflexivity and eliminated with the so-called J rule, which is a principle that allows to substitute equals for equals in the remaining part of the proof. Because proofs are hard to write using these primitives, Axi provides a convenient feature called \textit{chaining}, which allows reasoning with long chains of equations by writing down and justifying each step separately, similarly to how equality is usually handled with pen and paper. Besides equality, this feature also supports implications, biconditionals and other transitive relations.

Classical logic is realized using two primitives. Proof by contradiction allows to prove ($(\sim P \implies \bot) \implies P$) for any proposition $P$. The Axiom of Choice works as follows: while writing a program for the purpose of using it in a proof, we are allowed make use of a new primitive, \texttt{choose}, which turns a proof of $\exists x : A, P\ x$ into an element of type $A$. These programs that make use of the Axiom of Choice are not part of Axi's programming layer and cannot be run, but since their sole purpose is to be used as part of a proof, this does not pose any problems. Axi's handling of the Axiom of Choice might seem strange to mathematicians at first glance, but this presentation makes it convenient to use in practice together with the rest of the proof language. This approach to the Axiom of Choice was inspired by the Lean proof assistant.

\subsubsection{Rich interfaces}

Last but not least, the killer feature of the logic layer are \textit{rich interfaces}. These are the same interfaces from the programming layer that we already mentioned. However, what we did not mention yet is that in addition to declarations of constants and functions, they may also contain specifications, i.e. properties expected to be true of the declared constants and functions. When implementing such an interface, besides defining the constants and functions, we must also provide proofs of the specifications.

Rich interfaces let us precisely define what is expected of various pieces of the blockchain infrastructure. For example, we could define an interface of a DEX, with functions like placing orders, canceling orders and collecting tokens from filled orders, and specify these using properties like ``all orders that are not filled can be canceled'', ``the user can recover his money after canceling an order'', etc.

Any smart contract that attempts or pretends to implement a DEX would become much more credible and trustworthy if it implemented this DEX interface. This is because the interface is a relatively short and readable way of making promises and guarantees regarding the service's behaviour -- instead of having to read plenty of code, which moreover might be difficult to understand, the user only needs to search the code for a phrase like ``implements the DEX interface'' and read a few short logical specifications. Even though the latter task is still outside of most people's capabilities, it is vastly simpler than having to do your own audit of the code.

Moreover, this task can be offloaded to \textit{social learning}, i.e. after some time it might become common knowledge, a good practice, etc., that any purported DEX should implement the DEX interface. In such a scenario, the due diligence is reduced to just looking for the phrase ``implements the DEX interface'' in source code, which is in most people's reach.

\section{Interacting with the ledger in Axi}

In this appendix, we will see how Axi programs can interact with Khalani's ledger. First we'll discuss the big idea behind Axi's interaction with the ledger, which we might call the ``resource lifecycle''. Then we'll recall a simplified ledger model and look at the primitive (buil-in) types and functions that represent it in Axi. Finally we'll see a slightly longer example of an on-chain service, together with example transactions that set this service up and use it.

\subsection{The resource lifecycle}

A new resource type is created by a transaction that deploys a package of Axi code that contains the definition of this resource type. The only exception is Gas, the native coin used to pay transaction fees, which is built-in, so it already exists at the origin block. Because packages live on-chain forever and cannot be deleted, a resource type, once it comes to life, will also exist forever.

The life cycle of a particular resource, i.e. an element of some resource type, consists of three stages:

\begin{itemize}
  \item \textbf{Creation}. At some point the resource is created according to the rules set by the package that contains the definition of the resource's type. After creation, the resource is placed on the ledger in a cell owned by somebody.
  \item \textbf{Transaction(s)}. The resource might participate in transactions, if its owner wishes so. The owner might transfer his ownership of the resource freely, whereas programmatic interaction of the resource with other objects happens only according to the rules set by the defining package.
  \item \textbf{Destruction (optional)}. If the defining package permits it, the resource might at some point be destroyed. Examples of destructible resources are stablecoins that are backed by an off-chain reserve, where a coin gets destroyed when its off-chain collateral is removed from the treasury, and a wrapper for an asset bridged from another blockchain, which gets destroyed when the external asset ceases to be bridged.
\end{itemize}

The most important part of the lifecycle, in which most resources will spend almost all their life, is the phase when they can participate in transactions. An ordinary transaction (i.e. one which does not deploy a new package) consists of an arbitrary Axi program that transforms the state of the ledger, which we'll call the \textit{transaction body}, and of some metadata, most importantly the \textit{transaction originator}. This program, when executed, interacts with the ledger in three steps:

\begin{itemize}
  \item \textbf{Withdraw}. First, resources are ``withdrawn'' from the ledger into the program. The ledger's ownership model ensures the safety of this step, for example by making sure that the program cannot withdraw resources that don't belong to the transaction originator.
  \item \textbf{Transform}. Second, the program transforms its resources by performing various computations, exchanging the resources for other resources and moving them around, etc. The safety of this step is guaranteed by the type system. The resources cannot get unlawfully destroyed or copied because the type system keeps an eye on them, nonsensical operations cannot be performed on them, because the program is well-typed, they cannot get lost in the jungle of pattern matching, because exhaustiveness is checked, and so on.
  \item \textbf{Deposit}. Third, the program ``deposits'' the transformed resources back onto the ledger. The safety of this step is partially ensured by the type system, which checks that all resources held by the program are deposited back. However, since resources can be freely transferred, it's up to the transaction author to make sure that they are transferred to the correct address and not to somebody else.
\end{itemize}

\subsection{A simplified ledger model and Axi primitives}

Everything on-chain is stored in cells. Each cell has an address and some metadata, which won't concern us for now. There are two main kinds of cells: an \textit{object} is a cell that holds data or assets, whereas a \textit{package cell} is a cell that holds a single package of Axi code (possibly split into smaller units called \textit{modules}). Every object, besides its address and metadata, also has an (Axi) type, and stores exactly one value of that type.

Addresses, besides pointing to cells, also have a different, separate use: they may act as the identity of an \textit{account}. Accounts are used by external actors to interact with the blockchain, similarly to EOAs (Externally Owned Accounts) on Ethereum.

Objects are either \textit{owned} or \textit{shared}. Owned objects have exactly one owner who is an account. Shared objects do not have an owner. They are further subdivided into \textit{mutable shared objects}, which contain resources and can be interacted with by anybody, and \textit{immutable shared objects}, which contain data and can be read by anybody, but not modified.

The above code listing shows the most important primitives (built-ins) available in Axi. \code{Ledger} is an abstract resource type that represents the ledger. Every function that interacts with the ledger takes an argument of this type and also returns a value of this type. Linear types guarantee that the input ledger must be used exactly once and that a new version of the ledger must be returned. The transaction context will provide exactly one value of this type. Together these conditions guarantee that at any point during the execution of any program there is only a single value of this type, so the ledger cannot be ``forked'' in any way. Here \code{Address} is the type of addresses, including accounts.

\code{transfer} is a function that takes an account and a value of some type, together with a ledger, creates a new cell on the ledger that contains this value and belongs to that account, and returns the address of the cell together with a new ledger. This is the functions that on-chain programs will use to ``deposit'' resources back onto the ledger.

\code{receive} is a function that takes a cell address, together with a ledger, and returns an optional value of some type, together with the new ledger.
